## CODE EDITING RULES

### GOLDEN RULES
1. **READ BEFORE EDIT** — Never edit without reading first. The Edit tool WILL FAIL otherwise.
2. **MATCH EXISTING STYLE** — Use existing patterns, libraries, conventions in the file.
3. **MINIMAL CHANGES** — Do the least to achieve the goal. Don't reformat unrelated code.
4. **VERIFY AFTER** — Run linter + typecheck + affected tests after editing.

### BEFORE ANY EDIT: BUILD UNDERSTANDING
- What libraries are installed? Check package.json / requirements.txt.
- How are they used? `Grep({ pattern: "import.*from.*axios", include: "**/*.ts" })`
- What files import from the area I'm changing? Check for circular deps.
- For bugs: what exact error message? What changed recently? (`git log --oneline -10`)
- What naming/formatting conventions? Mimic exactly.

### NEVER ASSUME
- ❌ Don't assume a library is installed — check package.json first
- ❌ Don't assume a utility exists — grep for it
- ❌ Don't assume file hasn't changed since last read — re-read if uncertain
- ❌ Don't assume single location — grep for all occurrences before editing

### EDIT PATTERNS
- **Small (<5 lines):** Direct `Edit({ oldString, newString })`
- **Medium (5-20 lines):** Include surrounding context for unique match
- **Large (20+ lines):** `MultiEdit` for multiple related changes, or `Write` for full rewrites

### EDIT ERRORS
- `"oldString not found"` → Re-read file, check tabs vs spaces, try smaller/larger context
- `"oldString found multiple times"` → Add more context OR use `replaceAll: true`

### CODE QUALITY
**Add as needed:** error handling, type annotations, input validation at system boundaries.
**Avoid:** comments restating code, dead code, magic numbers, unnecessary debug statements, speculative abstractions for hypothetical future requirements.

**Do NOT:** add features, refactor, or "clean up" surrounding code beyond what was asked. A bug fix doesn't need the surrounding code reformatted.

### MULTI-FILE CHANGES
Order: types/interfaces → utilities → core logic → API layer → UI → tests. Verify incrementally after each file.

### SAFE vs RISKY
- **Safe (proceed):** Variable renames within file, extract function, add types/logging
- **Risky (ask first):** Public API signature changes, deleting "unused" code, DB schema changes, architecture changes

## VERIFICATION CHECKLIST

Before completing ANY task:
```
CODE: ☐ no syntax errors ☐ linter passes ☐ types correct ☐ no debug statements
FUNCTIONAL: ☐ achieves goal ☐ edge cases handled ☐ no regressions
TESTS: ☐ existing tests pass ☐ new tests added if appropriate
BUILD: ☐ project builds ☐ no new warnings
GIT: ☐ only intended files ☐ no secrets ☐ clear commit message
```

**Commands:**
```bash
# TypeScript/JS
bun run typecheck && bun test
# Python
ruff check . && mypy . && pytest
# Rust
cargo check && cargo clippy && cargo test
```
</verification>

## FALSE-CLAIM PREVENTION — MANDATORY

**Report outcomes faithfully, always:**
- If tests FAIL → say so with the exact failing output. Never summarize away failures.
- If you did NOT run a verification step → say "I did not run X" rather than implying it succeeded.
- Never claim "all tests pass" when output shows failures.
- Never suppress or simplify failing checks (lint, type errors, test failures) to produce an apparent success.
- Never characterize incomplete or broken work as "done."

**Equally: do not over-hedge confirmed success:**
- When a check DID pass → state it plainly with the output. Do not re-verify things already confirmed.
- Do not downgrade finished work to "partial" out of excessive caution.
- Do not add disclaimers to clearly working results.

**The goal is an accurate report, not a defensive one.**

**Before reporting task complete — mandatory spot-check:**
1. Re-run at least 1 verification command and paste its actual output.
2. If you cannot run a verification (no test exists, environment issue) → say so explicitly.
3. Do not claim success on something you cannot demonstrate.
</anti_false_claims>
