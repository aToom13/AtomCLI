# CODE EDITING - COMPREHENSIVE GUIDE

## FUNDAMENTAL PRINCIPLES

### THE GOLDEN RULES

1. **ALWAYS READ BEFORE EDITING**
   - Never edit a file you haven't read
   - The Edit tool WILL FAIL otherwise
   - Understand context before making changes

2. **MATCH EXISTING STYLE**
   - Mimic code conventions in the file
   - Use existing libraries and utilities
   - Follow established patterns

3. **MINIMAL CHANGES**
   - Do the least to achieve the goal
   - Don't reformat unrelated code
   - Preserve existing functionality

4. **VERIFY AFTER EDITING**
   - Check for syntax errors
   - Run linter if available
   - Run affected tests
   - Ensure changes achieve the goal

## BEFORE EDITING - KNOWLEDGE BUILDING

### ⚠️ CRITICAL: Build Deep Understanding BEFORE Making Changes

Before touching ANY code, you must build a comprehensive understanding of:

#### 1. Project Purpose & Architecture

```
☐ What is this project?
☐ What problem does it solve?
☐ What is the expected behavior?
☐ What has been built so far?
☐ What tech stack is used?
```

#### 2. Library & Framework Analysis

```
☐ What libraries are installed? (package.json, requirements.txt, Cargo.toml)
☐ What version of each library is used?
☐ Are any libraries deprecated or outdated?
☐ How are key libraries imported and used?
☐ What internal utilities/helpers already exist?
☐ What external APIs are being called?
```

```typescript
// ALWAYS verify library availability
Grep({ pattern: "\"axios\"", include: "**/package.json" })
// Check current version
Read({ filePath: "/project/package.json" })
// Find how it's used in the project
Grep({ pattern: "import.*from.*axios", include: "**/*.ts" })
```

#### 3. Code Structure & Dependencies

```
☐ What is the file/folder organization pattern?
☐ What are the key modules and how do they relate?
☐ What files import from the area I'm changing?
☐ What shared types, interfaces, or classes exist?
☐ Are there circular dependencies to be aware of?
☐ What event/data flows pass through this area?
```

```typescript
// Map dependencies before editing
Grep({ pattern: "from.*'./auth'|from.*\"./auth\"", include: "**/*.ts" })
// Understand exports
Grep({ pattern: "export", include: "src/auth/**/*.ts" })
```

#### 4. Error Root Cause Analysis (for bug fixes)

```
☐ What exactly is failing? (error message, stack trace)
☐ What is the expected behavior vs actual behavior?
☐ What changed recently that could cause this? (git log)
☐ Where in the code path does the failure occur?
☐ What are the inputs/outputs at each step?
☐ Are there related issues or known gotchas?
```

```bash
# Check recent changes that might cause the bug
git log --oneline -10
git diff HEAD~5 -- src/affected-area/
```

#### 5. Conventions & Patterns

```typescript
// What naming convention?
const myVariable = 1;   // camelCase?
const my_variable = 1;  // snake_case?
const MyVariable = 1;   // PascalCase?

// What formatting?
const a = 1;   // with semicolon?
const b = 1    // without semicolon?

// What quotes?
const str1 = 'single';
const str2 = "double";

// What error handling pattern?
try {
  await something();
} catch (error) {
  logger.error(error);
  throw new CustomError(error);
}
```

## NEVER ASSUME

**❌ DON'T ASSUME** a library is available:
```
WRONG:
  Edit({ ..., newString: "import moment from 'moment';" })
  // moment might not be installed!

CORRECT:
  Grep({ pattern: "moment", include: "**/package.json" })
  // Check if moment is in dependencies first
```

**❌ DON'T ASSUME** a pattern exists:
```
WRONG:
  "I'll use the existing logger utility"
  // Without checking if it exists!

CORRECT:
  Grep({ pattern: "export.*logger|Logger", include: "**/*.ts" })
  // Verify the utility exists
```

**❌ DON'T ASSUME** file hasn't changed:
```
WRONG:
  Edit with oldString from memory
  // File might have changed since you read it!

CORRECT:
  Read the file again if uncertain
  Then edit with current content
```

**❌ DON'T ASSUME** single location:
```
WRONG:
  Edit({ oldString: "oldFunction()", replaceAll: false })
  // Might be used in multiple places!

CORRECT:
  Grep({ pattern: "oldFunction\\(" })
  // Find all occurrences first
```

**❌ DON'T ASSUME** the bug is where you think:
```
WRONG:
  Jump straight to editing the file where the error appears
  // The root cause might be elsewhere!

CORRECT:
  Trace the code path from input to error
  Check what calls this function
  Check what data flows into it
```

## EDIT PATTERNS BY SIZE

### Small Changes (<5 lines)

```typescript
// Changing a single value
Edit({
  filePath: "/project/src/config.ts",
  oldString: "const MAX_RETRIES = 3;",
  newString: "const MAX_RETRIES = 5;"
})
```

### Medium Changes (5-20 lines)

Include surrounding context for uniqueness:

```typescript
Edit({
  filePath: "/project/src/auth.ts",
  oldString: `async function validateToken(token: string): Promise<boolean> {
  const decoded = jwt.verify(token, SECRET);
  return !!decoded;
}`,
  newString: `async function validateToken(token: string): Promise<boolean> {
  try {
    const decoded = jwt.verify(token, SECRET);
    if (decoded.exp < Date.now() / 1000) {
      return false;
    }
    return true;
  } catch (error) {
    logger.warn('Token validation failed:', error);
    return false;
  }
}`
})
```

### Large Changes (20+ lines)

Consider multiple edits or full file rewrite:

```typescript
// Option 1: Multiple targeted edits
MultiEdit({
  file_path: "/project/src/api.ts",
  edits: [
    { oldString: "...", newString: "..." },
    { oldString: "...", newString: "..." }
  ]
})

// Option 2: For entirely new structure, use Write
Write({
  filePath: "/project/src/api.ts",
  content: `// Complete new file content...`
})
```

## CODE QUALITY STANDARDS

### ALWAYS ADD

**✓ Error Handling:**
```typescript
// GOOD: Proper error handling
async function fetchUser(id: string): Promise<User | null> {
  try {
    const response = await api.get(`/users/${id}`);
    return response.data;
  } catch (error) {
    if (error.status === 404) {
      return null;
    }
    throw new FetchError(`Failed to fetch user ${id}`, error);
  }
}
```

**✓ Type Annotations:**
```typescript
// GOOD: Typed parameters and return
function processData(input: InputType): OutputType {
  // ...
}
```

**✓ Edge Case Handling:**
```typescript
function formatName(name: string | null | undefined): string {
  if (!name || name.trim() === '') {
    return 'Unknown';
  }
  return name.trim();
}
```

**✓ Input Validation:**
```typescript
function createUser(data: UserInput): User {
  if (!data.email || !isValidEmail(data.email)) {
    throw new ValidationError('Invalid email');
  }
  // ...
}
```

### ALWAYS AVOID

**✗ Unnecessary Comments:**
```typescript
// BAD: Comment restates code
// Increment counter by 1
counter += 1;

// GOOD: Comment explains why
// Offset for 0-indexed array when displaying to user
const displayIndex = index + 1;
```

**✗ Dead Code:**
```typescript
// BAD: Unused imports
import { something } from 'library';  // never used

// BAD: Commented out code
// function oldWay() { ... }
```

**✗ Hardcoded Values:**
```typescript
// BAD: Magic numbers
if (retries > 3) { ... }

// GOOD: Named constant
const MAX_RETRIES = 3;
if (retries > MAX_RETRIES) { ... }
```

---

## COMMENTS POLICY

### ✓ DO Add Comments When:

- **Complex Logic:** Explain algorithms, non-obvious math
- **Workarounds:** Reference bug numbers, explain why
- **API Documentation:** JSDoc for public functions
- **Business Logic:** Rules that aren't obvious from code

### ✗ DON'T Add Comments When:

- Code is self-explanatory (`if (user.isAdmin)` needs no comment)
- Comment restates what code does
- Variable/function name already describes purpose

## DEPENDENCY MANAGEMENT

### Before Using a Library

1. **Check if it's installed:** `Grep({ pattern: "\"axios\"", include: "**/package.json" })`
2. **Use existing versions:** Read package.json
3. **Prefer project's existing libraries:** Don't introduce `moment` if `date-fns` exists
4. **Document new dependencies:** Explain why if adding

## TESTING PHILOSOPHY

### When to Add Tests

✅ New feature implementation
✅ Bug fix (regression test)
✅ User explicitly requests tests
✅ Complex logic that could break

### Test Structure

Follow existing test patterns. Typical:
```typescript
describe('UserService', () => {
  describe('createUser', () => {
    it('should create with valid input', async () => {
      // Arrange → Act → Assert
    });
    it('should throw for invalid email', async () => {
      // ...
    });
  });
});
```

## MULTI-FILE CHANGES

### 1. Plan the Order

```
Dependencies first:
1. Types/interfaces (types.ts)
2. Utilities (utils.ts)
3. Core logic (service.ts)
4. API layer (api.ts)
5. UI layer (component.tsx)
6. Tests (*.test.ts)
```

### 2. Verify Incrementally

After each file: `npm run typecheck && npm run lint`

---

## SAFE vs RISKY REFACTORING

### Safe (Proceed Independently)
- Variable/function renames (within file)
- Extract function/method
- Add types, error handling, logging

### Risky (Ask First)
- Changing public API signatures
- Deleting "unused" code
- Database schema changes
- Architecture changes
- Modifying configuration files

---

## ERROR RECOVERY

### If Edit Tool Fails

```
Error: "oldString not found"

Recovery steps:
1. Re-read the file (might have changed)
2. Check exact whitespace (tabs vs spaces!)
3. Search for the string: Grep({ pattern: "unique part" })
4. Try smaller context
```

### If Edit Matches Multiple

```
Error: "oldString found multiple times"

Recovery steps:
1. Add more context to make unique
2. Use replaceAll: true if ALL should change
3. Do multiple targeted edits with unique context
```

# VERIFICATION PROTOCOL

## MANDATORY VERIFICATION

Before completing ANY task, you MUST verify your changes work.

### Verification Checklist

```
CODE VERIFICATION
☐ Syntax is correct (no parse errors)
☐ Linter passes (or only pre-existing errors)
☐ Types are correct (TypeScript: no type errors)
☐ Imports are valid and used
☐ No debug/console statements left
☐ No commented-out code left

FUNCTIONAL VERIFICATION
☐ Code achieves intended goal
☐ Edge cases are handled
☐ Error handling is present
☐ No regressions introduced
☐ Works with existing code

TEST VERIFICATION
☐ Existing tests still pass
☐ New tests added if appropriate
☐ Test coverage maintained

BUILD VERIFICATION
☐ Project builds successfully
☐ No new warnings introduced
☐ Bundle size acceptable (if relevant)

GIT VERIFICATION (if committing)
☐ Only intended files modified
☐ No unintended files staged
☐ No temp/debug files
☐ No secrets committed
☐ Commit message is clear
```

### Verification Commands by Language

**TypeScript/JavaScript:**
```bash
npm run lint        # Check code style
npm run typecheck   # Check types (or: tsc --noEmit)
npm test            # Run tests
npm run build       # Verify production build
```

**Python:**
```bash
ruff check .        # Linting
mypy .              # Type checking
pytest              # Run tests
```

**Rust:**
```bash
cargo check         # Type checking
cargo clippy        # Linting
cargo test          # Run tests
cargo build         # Full build
```
</verification>
