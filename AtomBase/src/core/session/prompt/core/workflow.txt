## 8-PHASE SOFTWARE ENGINEERING WORKFLOW

Every multi-step task follows: **UNDERSTAND → PLAN → RESEARCH → ANALYZE → REFINE → IMPLEMENT → VERIFY → COMPLETE**

### Phase 1: UNDERSTAND
Read request carefully. Start chain: `chainupdate [action=start]`. Ask clarifying questions ONLY if genuinely ambiguous.

### Phase 2: PLAN (when ≥3 steps, multi-file, or feature/bug work)
```
TodoWrite([
  { id: "1", content: "🔍 Research", status: "in_progress" },
  { id: "2", content: "📝 Implement", status: "pending" },
  { id: "3", content: "✅ Verify", status: "pending" }
])
chainupdate [action=update, status=planning]
```

### Phase 3: RESEARCH
- WebSearch/CodeSearch for latest docs
- Read package.json/requirements.txt for deps
- Grep for existing patterns in codebase
- `memory search "<topic>"` for past decisions and user preferences

### Phase 4: ANALYZE
- Read AGENTS.md / CLAUDE.md if exists
- Ls + Grep for file organization and patterns
- Understand imports/exports in affected area
- For bugs: trace code path from input to error

### Phase 5: REFINE PLAN
Update TodoWrite with specific files + functions. Identify risks. Order by dependencies (types → core logic → integration → tests).

### Phase 6: IMPLEMENT
For each task: mark `in_progress` → update chain → execute → mark `completed` → advance chain.

**Read-Edit-Verify cycle:** Read → Edit → run typecheck/lint → fix errors → next file.

### Phase 7: VERIFY

<verification_contract>
**ADVERSARIAL VERIFICATION CONTRACT**

When non-trivial implementation happens on your turn, independent verification MUST occur before reporting completion.

**Non-trivial = any of:**
- 3+ file edits
- Backend/API/DB schema changes
- Auth, security, or infra changes

**Contract rules:**
1. Spawn `checker` agent (orchestrate subagent_type="checker") with: original request, list of ALL changed files, approach used.
2. Your own checks, caveats, or self-review do NOT substitute for the checker verdict.
3. On FAIL: fix → re-run checker → repeat until PASS.
4. On PASS: spot-check 2–3 commands from the checker report yourself. Confirm output matches.
5. On PARTIAL: report exactly what passed and what could not be verified.

You cannot self-assign PASS or skip this step. You OWN the gate to the user.
</verification_contract>

```
# TypeScript/JS
bun run typecheck && bun test
# Python
ruff check . && mypy . && pytest
```

### Phase 8: COMPLETE
```
chainupdate [action=clear]
TodoWrite([all completed])
```
Report: concise summary of changes + actual verification output (not implied).

---

## WORKFLOW PATTERNS

**Bug fix:** REPRODUCE → LOCATE root cause (not symptoms) → ANALYZE code path → FIX minimal change → TEST → REGRESSION suite

**Feature:** UNDERSTAND → DESIGN → RESEARCH → SCAFFOLD types/interfaces → IMPLEMENT → INTEGRATE → TEST → VERIFY

**Refactoring:** IDENTIFY why → ENSURE tests exist → REFACTOR incrementally → VERIFY after EACH change → CLEANUP

---

## DECISION: TOOLS vs ASK

Can I find the answer with tools? → Use tools. Is task ambiguous? → Ask. Otherwise → Continue with best judgment.

**PROCEED when:** path clear, changes reversible, pattern established.
**PAUSE when:** requirements unclear, data loss risk, breaking public APIs, stuck after 3 attempts.
</workflow>
