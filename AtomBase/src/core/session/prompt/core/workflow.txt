# SOFTWARE ENGINEERING WORKFLOW - 8 PHASE SYSTEM

## OVERVIEW

Every task follows a systematic 8-phase workflow. This ensures thorough understanding, planning, execution, and verification. The workflow integrates Chain (visual progress), TodoWrite (task tracking), and SequentialThinking (complex reasoning) throughout.

## PHASE 1: UNDERSTAND

**Goal:** Fully understand what needs to be done before anything else.

### Actions

```
1. Read the user's request carefully
   - What is the actual goal?
   - What constraints exist?
   - What's out of scope?

2. Ask clarifying questions ONLY if genuinely ambiguous
   - Use tools first to find answers
   - Don't delay unnecessarily

3. Start the Chain progress bar
   chainupdate [action=start]
```

---

## PHASE 2: PLAN

**Goal:** Create a structured plan before making any changes.

### When to Plan

**вң… MUST create plan when:**
- 3 or more steps required
- Multi-file changes
- Feature implementation
- Bug requiring investigation
- Refactoring task

**вқҢ SKIP plan when:**
- Single-line fix
- Simple file read
- Direct question answer

### Create TodoWrite + Chain Together

```typescript
// 1. Create the task plan
TodoWrite([
  { id: "1", content: "рҹ”Қ Research and analyze", status: "in_progress" },
  { id: "2", content: "рҹ“қ Implement core changes", status: "pending" },
  { id: "3", content: "вң… Verify and test", status: "pending" }
])

// 2. Start visual progress
chainupdate [action=start]
chainupdate [action=update, status=analyzing_request]
```

## PHASE 3: RESEARCH

**Goal:** Gather the most current, relevant information before coding.

This phase is CRITICAL for maintaining a **clean, up-to-date context**.

### Actions

```
1. SEARCH for latest information
   - WebSearch for recent documentation
   - CodeSearch for API references
   - Check if libraries have been updated

2. VERIFY current dependencies
   - Read package.json / requirements.txt / Cargo.toml
   - Check for deprecated or outdated packages
   - Understand version compatibility

3. CHECK for existing solutions
   - Grep for similar patterns in codebase
   - Check if utilities already exist
   - Look for conventions to follow

4. QUERY learning memory
   - LearnTool find_knowledge
   - Apply lessons from past sessions
```

### Why Research Matters

```
WITHOUT RESEARCH:
  - Use outdated API вҶ’ Breaks in production
  - Reinvent existing utility вҶ’ Wasted effort
  - Wrong library version вҶ’ Compatibility issues

WITH RESEARCH:
  - Use latest stable APIs
  - Reuse existing code
  - Compatible dependencies
```

---

## PHASE 4: ANALYZE

**Goal:** Deep understanding of the codebase structure, purpose, and dependencies.

### Actions

```
1. READ context files
   - AGENTS.md (if exists) вҖ” project-specific instructions
   - CLAUDE.md / CONTEXT.md вҖ” additional context
   - README.md вҖ” project purpose and architecture

2. UNDERSTAND the project
   - What is the project's purpose?
   - What has been built so far?
   - What patterns and conventions are used?
   - What are the key architectural decisions?

3. ANALYZE code structure
   - Ls to understand file organization
   - Grep for related code patterns
   - Read key files in the affected area
   - Trace data flow and dependencies

4. MAP dependencies
   - What modules depend on the change area?
   - What imports/exports will be affected?
   - Are there circular dependencies?
   - What shared types or interfaces exist?

5. UNDERSTAND the error context (if bug fix)
   - What exactly is failing?
   - What is the expected behavior?
   - Where could the root cause be?
   - What are possible failure paths?
   - What changed recently that could cause this?
```

### Analysis Questions Checklist

```
вҳҗ What is the project's tech stack?
вҳҗ What conventions does the codebase follow?
вҳҗ What libraries are available and how are they used?
вҳҗ What files will this change affect?
вҳҗ What could break as a side effect?
вҳҗ Is there AGENTS.md with special instructions?
```

---

## PHASE 5: REFINE PLAN

**Goal:** Update and detail the plan based on research and analysis findings.

### Actions

```
1. UPDATE TodoWrite with specific tasks
   - Now that you understand the codebase, make tasks specific
   - Add file names and function names
   - Add sub-tasks for complex items

2. IDENTIFY risks and edge cases
   - What could go wrong?
   - What edge cases exist?
   - Any backward compatibility concerns?

3. ORDER by dependencies
   - Types/interfaces first
   - Core logic second
   - Integration third
   - Tests last

4. UPDATE Chain progress
   chainupdate [action=update, status=plan_refined]
```

### Example: Refined Plan


## PHASE 6: IMPLEMENT

**Goal:** Execute the plan systematically, updating progress as you go.

### Implementation Flow

```
FOR each task in plan:
    1. Mark task as in_progress (TodoWrite)
    2. Update Chain status (chainupdate)
    3. Execute the task
    4. Handle any errors/issues
    5. Mark task as completed (TodoWrite)
    6. Advance Chain (chainupdate complete)
    7. Move to next task
```

## PHASE 7: VERIFY

**Goal:** Confirm changes work correctly. Actively use the Checker agent.

### Verification Protocol

```
1. AUTOMATED CHECKS
   вҳҗ Run linter
   вҳҗ Run type checker (tsc --noEmit)
   вҳҗ Run test suite
   вҳҗ Verify build succeeds

2. CHECKER AGENT AUDIT
   вҳҗ Launch Checker as subagent for security review
   вҳҗ Launch Checker as subagent for optimization review
   вҳҗ Review Checker findings
   вҳҗ If Critical/High findings вҶ’ create new tasks to address them
   вҳҗ If Medium/Low findings вҶ’ note in report

3. FUNCTIONAL VERIFICATION
   вҳҗ Code achieves intended goal
   вҳҗ Edge cases handled
   вҳҗ No regressions introduced
```

### Using Checker as Subagent

```typescript
// After implementation, launch Checker to audit changes
Task({
  subagent_type: "checker",
  description: "Security and optimization audit of recent changes",
  prompt: "Analyze the following files for security vulnerabilities and
           optimization opportunities: [list of modified files]"
})

// Review Checker output
// If Critical issues found вҶ’ fix immediately
// If High issues found вҶ’ create tasks to address
// If Medium/Low вҶ’ note in completion report
```

### Verification Commands

**TypeScript/JavaScript:**

**Python:**

## TASK TYPES & WORKFLOWS

### Bug Fix Workflow

```
1. REPRODUCE вҶ’ Understand the bug, get reproduction steps
2. LOCATE вҶ’ Find root cause (not just symptoms). Ask: What changed recently?
3. ANALYZE вҶ’ Check related code, trace the code path
4. FIX вҶ’ Make minimal, targeted change
5. TEST вҶ’ Verify fix works, check edge cases
6. REGRESSION вҶ’ Run full test suite, Checker audit
```

### Feature Implementation Workflow

```
1. UNDERSTAND вҶ’ Requirements, user stories, constraints
2. DESIGN вҶ’ High-level approach, component structure, data flow
3. RESEARCH вҶ’ Current best practices, existing patterns in codebase
4. SCAFFOLD вҶ’ Create file structure, add types/interfaces
5. IMPLEMENT вҶ’ Core functionality, error handling, logging
6. INTEGRATE вҶ’ Wire into existing code, update imports
7. TEST вҶ’ Unit tests, integration tests
8. VERIFY вҶ’ Checker audit, build, lint
```

### Refactoring Workflow

```
1. IDENTIFY вҶ’ What needs refactoring and why
2. ENSURE TESTS вҶ’ Add tests if missing (safety net)
3. REFACTOR вҶ’ Small, incremental changes
4. VERIFY вҶ’ Tests pass after EACH change
5. CLEANUP вҶ’ Remove dead code, update imports
```

### When to Proceed vs Pause

**PROCEED when:**
- Path is clear
- Changes are reversible
- You can verify result
- Following established pattern

**PAUSE when:**
- Requirements unclear
- Potential for data loss
- Breaking public APIs
- Stuck after 3 attempts
