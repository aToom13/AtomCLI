## GIT SAFETY RULES

**NEVER without explicit user permission:**
- `git push --force` / `--force-with-lease`
- `git reset --hard`
- `git clean -fd` / `-fdx`
- `git rebase` on shared branches
- `git commit --amend` (if already pushed)
- `git branch -D` / `git push origin --delete`
- `git commit --no-verify` / `git push --no-verify`
- Modifying `git config user.*` or remote URLs

**Safe without asking:**
- All read-only ops: `git status`, `git diff`, `git log`, `git show`, `git branch`, `git remote -v`
- `git checkout`/`switch`, `git add <specific-file>`, `git add -p`, `git stash`, `git fetch`, `git pull`, `git push` (non-force), `git commit`

**Before every commit:**
1. `git status` + `git diff --cached` — review what's staged
2. Check for secrets: `git diff --cached | grep -iE "(password|secret|key|token|auth)"`
3. Only stage specific files — avoid `git add .`

**Never commit:** API keys, passwords, tokens, `.env` files, `node_modules/`, `dist/`, `build/`, `.DS_Store`

**Commit workflow (safe order):**
1. Run parallel: `git status` (never `-uall`) AND `git diff` AND `git log --oneline -5`
2. Draft commit message focused on WHY, not what
3. Stage specific files: `git add <file1> <file2>`
4. Commit using HEREDOC to preserve formatting:
```bash
git commit -m "$(cat <<'EOF'
type(scope): description

Body explaining why this change was made.
EOF
)"
```
5. Run `git status` afterwards to verify success

**CRITICAL — Pre-commit hook failure:**
If a pre-commit hook FAILS, the commit did NOT happen. NEVER use `--amend` afterwards — that would modify the PREVIOUS commit. Instead: fix the issue → re-stage → create a NEW commit.

**Conflict resolution:** Don't panic. Understand both sides. Remove ALL conflict markers. Test after resolving.

**Commit message format:** `<type>(<scope>): <description>` — Types: feat, fix, refactor, docs, test, chore, style, perf
